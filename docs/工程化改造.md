# 工程化改造方案

> 本文档记录 Quant 项目的架构诊断、改造目标、落地步骤与版本管理规范。
> 改造原则：**行为不变、小步快跑、每步可验证**。

---

## 一、背景

当前代码功能可用（数据采集 + 因子计算已打通），但存在结构性耦合问题，
不利于后续接入模型训练、回测、实盘执行等模块。本文档规划一次工程化改造，
目标是让代码「成熟、可协作、可上 GitHub」。

---

## 二、现状诊断

### 1. 配置散落，无单一配置源

- `TRADE_UNIVERSE`、`STALE_DAYS`、路径、批大小等业务参数硬编码在
  [quant/fetcher.py](../quant/fetcher.py) 中，与 IO 逻辑耦合。
- [fetcher.py:_load_env](../quant/fetcher.py) 手搓 `.env` 解析，
  但 [requirements.txt](../requirements.txt) 已声明 `python-dotenv`——
  依赖装了却没用，重造轮子且易出 bug。
- `.env` / 环境变量 / 硬编码三处配置优先级混乱。

**影响**：换标的池、调缓存策略都要改数据模块代码；配置无法在不同环境（开发/实盘）间切换。

### 2. 模块职责过载，backend 用 if/elif 分发

- [fetcher.py](../quant/fetcher.py) 单文件塞了：env 读取、代理构造、yfinance 后端、
  AV 后端、缓存、完整性检查、CLI。
- backend 选择是 `if backend == "yfinance" ... elif ...`，加新数据源
  （如 akshare、Exness MT5 报价）就要改分发函数——破坏开闭原则。

**影响**：后续接 Exness 实盘报价时，无法干净地插入新 backend。

### 3. CLI 入口脆弱且重复

- [data_fetcher.py](../data_fetcher.py) 和 [factor_engine.py](../factor_engine.py)
  都用裸 `sys.argv` 解析，手写 `--force`/`--max`/`--cache`，无 `--help`、无类型校验。
- 两个脚本各自 `logging.basicConfig`——日志配置重复。
- `factor_engine.py --cache` 语义反直觉（实为「不刷新」）。

### 4. 重复验证

- [factor_engine.py:58](../factor_engine.py) 调 `validate_factors()`，
  但 `build_factor_panel(validate=True)` 内部已调过一次——同一份数据验证两遍。

### 5. 包不可安装、无测试

- 无 `pyproject.toml`，必须 `cd` 到项目根才能 `import quant`
  （[操作流程.md](操作流程.md) 已把这个坑写进 FAQ）。
- [quant/__init__.py](../quant/__init__.py) 空文件，无 `__version__`、无顶层导出。
- [test/](../test/) 全是一次性脚本，非 pytest 用例——8 个因子纯函数最该有单测却没写。

### 6. 数据目录扁平

- `data/` 里 raw 行情、`factor_validation.csv` 混在一起，无 `raw/`/`factors/`/`reports/` 分层。

## 三、目标架构

```
quant/
├── config.py              # 新：集中配置（标的池、路径、阈值），接入 python-dotenv
├── io/
│   ├── __init__.py
│   ├── base.py            # 新：DataBackend 协议（fetch → MultiIndex DataFrame）
│   ├── yfinance_backend.py
│   ├── alpha_vantage.py
│   ├── mt5_backend.py     # 预留：Exness 实盘报价（MetaTrader5）
│   └── cache.py           # parquet 读写 + stale 判断
├── factors/
│   ├── __init__.py
│   ├── momentum.py        # 拆分：动量因子
│   ├── meanreversion.py   # 均值回归因子
│   ├── volatility.py      # 波动率因子
│   └── engine.py          # 组装器 + 验证
├── integrity.py           # 数据完整性检查（从 fetcher 拆出）
└── cli.py                 # 新：click/typer 统一 CLI（fetch / factors 子命令）

pyproject.toml             # 新：可 pip install -e .，不再依赖 cwd
tests/                     # 新：pytest，因子纯函数单测
data/{raw,factors,reports}/  # 分层
configs/                   # 新：策略/回测 YAML 配置（后续阶段）
```

## 四、落地优先级

每步独立可验证，行为不变。

| 阶段 | 内容 | 验证标准 |
|---|---|---|
| **P1** | 抽 `config.py`，接入 `python-dotenv`，移除手搓 `_load_env`；迁移 `TRADE_UNIVERSE` 等常量 | 现有脚本输出与改造前逐字节一致 |
| **P2** | `DataBackend` 协议 + 拆 `io/`；yfinance/AV 各成文件；`fetch_all_data` 改注册表分发；预留 `MT5Backend` 接口 | mock backend 单测通过；真实 yfinance 仍能跑 |
| **P3** | `pyproject.toml`（`pip install -e .`）+ `click` 统一 CLI（`quant fetch`/`quant factors`）+ pytest 骨架（8 个因子纯函数单测） | `pytest` 全绿；`quant fetch --max 3` 与旧脚本等价 |
| **P4** | 数据目录分层 `raw/factors/reports`；因子拆分到 `factors/` 子包；拆 `integrity.py` | 因子面板输出 shape 不变 |

> P1 是最低风险起点，建议先做。P1–P3 可在推 GitHub 前完成，P4 随模型训练阶段一起做。

## 五、版本管理

### 5.1 Git 初始化清单（推 GitHub 前必做）

- [x] [.gitignore](../.gitignore)：排除 `.env`、`.venv/`、`data/` 内容、`__pycache__/`、IDE 文件
- [x] [.env.example](../.env.example)：配置模板（不含真实密钥）
- [x] `git init`
- [ ] 首次提交（见下方命令）

首次提交命令：

```bash
git add .
git status              # 确认 .env、data/*.parquet 未被加入
git commit -m "chore: initial commit — data fetcher + factor engine"
```

> ⚠️ 提交前务必 `git status` 确认 `.env` **不在**暂存区。若误提交密钥，
> 参考 [GitHub 移除敏感数据](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository)。

### 5.2 分支策略（Git Flow 简化版）

| 分支 | 用途 | 命名 |
|---|---|---|
| `main` | 稳定可发布版本 | `main` |
| `develop` | 集成分支 | `develop` |
| feature | 新功能 | `feat/<短描述>`，如 `feat/config-center` |
| fix | bug 修复 | `fix/<短描述>` |
| refactor | 重构 | `refactor/<短描述>` |

个人项目可简化为 `main` + feature 分支，PR 合并后删除 feature 分支。

### 5.3 提交规范（Conventional Commits）

```
<type>(<scope>): <subject>

<body>
```

- **type**：`feat`(新功能) `fix`(修复) `refactor`(重构) `docs`(文档) `test`(测试)
  `chore`(构建/工具) `perf`(性能)
- **scope**（可选）：`fetcher` `engine` `config` `cli` 等
- **subject**：祈使句，小写开头，不加句号

示例：

```
feat(config): 抽取配置中心，接入 python-dotenv
fix(engine): 修复 RSI 持续下跌时返回 NaN 的问题
docs: 新增工程化改造方案文档
refactor(io): 拆分 DataBackend 协议
```

### 5.4 版本号（语义化版本 + tag）

```
v<MAJOR>.<MINOR>.<PATCH>
```

- MAJOR：不兼容的架构变更
- MINOR：向后兼容的新功能
- PATCH：向后兼容的修复

里程碑打 tag：

```bash
git tag -a v0.1.0 -m "数据采集 + 因子计算 MVP"
git push origin v0.1.0
```

建议版本规划：

| 版本 | 里程碑 |
|---|---|
| `v0.1.0` | 数据采集 + 因子计算（现状） |
| `v0.2.0` | 工程化重构完成（P1–P3） |
| `v0.3.0` | 模型训练（LightGBM + Walk-Forward） |
| `v0.4.0` | 回测验证（vectorbt） |
| `v0.5.0` | 实盘执行（MT5 + Exness） |
| `v1.0.0` | 全链路稳定版 |

### 5.5 CHANGELOG

每次发版在仓库根维护 `CHANGELOG.md`（Keep a Changelog 格式），
记录 Added / Changed / Fixed / Removed。改造落地后补建。
